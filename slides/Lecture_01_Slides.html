1. The Small Files Problem

The Gap Between Big Data and AI

Spark produces output as distributed partitions — millions of small files across HDFS nodes.

AI training needs sequential, high-throughput reads — GPUs process data orders of magnitude faster than storage can deliver it file-by-file.

Core tension: Spark optimizes for parallel random access. Deep learning optimizes for sequential streaming.

What Happens Per File Access?

Every individual file request incurs a seek penalty:

StepOverhead
DNS Resolution~1–50 ms
TCP Handshake (SYN/SYN-ACK/ACK)~1–10 ms
TLS Negotiation (HTTPS)~5–30 ms
S3 API Auth (AWS Sig V4)~5–20 ms
Disk/SSD Seek~0.1–10 ms
Total Tseek~12–120 ms

These costs are paid per file, regardless of file size.

The I/O Latency Equation

Baseline: Independent Files

$$T_{total} = N \times \left(T_{seek} + \frac{S_{file}}{B_{network}}\right)$$

With CIFAR-10: $N=60{,}000$, $T_{seek}=50\text{ms}$, $S_{file}=3\text{KB}$, $B=1\text{Gbps}$

Seek penalty alone: $60{,}000 \times 0.05\text{s} = \mathbf{3{,}000\text{s}}$ — 50 minutes of GPU idle time!

The Solution: Contiguous Sharding

Amortized I/O with Tar Archives

$$T_{total} \approx T_{seek\_archive} + \frac{N \times S_{file}}{B_{network}}$$

Pack 1,000 images per .tar shard → only 50 seeks instead of 60,000.

Before (60K files)

  • 60,000 HTTP roundtrips
  • 3,000s seek overhead
  • GPU utilization: ~0%

After (50 shards)

  • 50 HTTP connections
  • 2.5s seek overhead
  • GPU utilization: ~95%+

2. Object Storage Fundamentals

Why Not Just Use a Filesystem?

Filesystem (HDFS, ext4)

  • Hierarchical directories
  • Mutable files (append, overwrite)
  • POSIX semantics (open/read/seek/close)
  • Scales to ~millions of files

Object Storage (S3)

  • Flat namespace (buckets + keys)
  • Immutable objects (replace only)
  • HTTP API (GET / PUT / DELETE)
  • Scales to trillions of objects

S3 is the de facto standard for AI training data. AWS, GCP (GCS), Azure (Blob) all expose S3-compatible APIs.

MinIO: S3 on Your Laptop

MinIO is an open-source, S3-compatible object store that runs as a single binary.

Why MinIO for this course?
  • Identical API to AWS S3 — same boto3 code works on both
  • Runs locally — no cloud account, no cost, no network dependency
  • Web console at http://127.0.0.1:9000

3. Data Serialization with WebDataset

Why .tar and not .zip?

ZIP

  • Central directory at end of file
  • Must seek to EOF before reading
  • ❌ Not streamable over HTTP

TAR

  • Sequential header + data blocks
  • Read from byte 0, no index needed
  • ✅ Perfectly streamable over HTTP
WebDataset stores each sample as a group of files sharing a common key:
000042.png (image) + 000042.cls (label) → one training sample

Shard Generation

# scripts/01_prepare_shards.py (simplified)
import torchvision, webdataset as wds

dataset = torchvision.datasets.CIFAR10(root='./data',
                                        train=True, download=True)

with wds.ShardWriter("cifar-train-%06d.tar",
                      maxcount=1000) as sink:
    for i, (img, label) in enumerate(dataset):
        sink.write({
            "__key__": f"{i:06d}",  # Groups related files
            "png": img,             # Auto-serialized to PNG bytes
            "cls": label            # Stored as text file
        })

Output: 50 shards × ~3 MB each = 150 MB total (vs. 60,000 individual files)

Uploading Shards to S3

import boto3, glob

s3 = boto3.client('s3',
    endpoint_url='http://localhost:9000',
    aws_access_key_id='minioadmin',
    aws_secret_access_key='minioadmin')

# Create bucket + set public-read policy
s3.create_bucket(Bucket="cifar-streaming")

# Upload each shard as an S3 object
for f in glob.glob("cifar-train-*.tar"):
    s3.upload_file(f, "cifar-streaming", f)
Same code works with AWS S3 — just remove endpoint_url and use real IAM credentials.

4. Streaming Pipeline

From Network → GPU, Zero Local Disk

import webdataset as wds

url = "http://127.0.0.1:9000/cifar-streaming/" \
      "cifar-train-{000000..000049}.tar"

dataset = (
    wds.WebDataset(url, shardshuffle=False)
    .decode("rgb")           # PNG bytes → float32 array
    .to_tuple("png", "cls")  # Package as (image, label)
)

Each pipeline stage is lazy — data flows on-demand as you iterate, never fully materializing in RAM.

The WebDataset Pipeline Stages

StageInputOutput
WebDataset(url)URL patternRaw tar byte streams
.decode("rgb")PNG byte buffernumpy float32 (H,W,3) [0,1]
.to_tuple("png","cls")Dict of decoded fields(image_array, label_int) tuples
.shuffle(K)Ordered streamPseudo-random stream (buffer K)
.batched(N)Individual samplesBatches of N samples

All stages are composable and lazy — no intermediate copies to disk.

Verifying the Stream

We visualize 16 streamed images in a 4×4 grid using matplotlib:

import matplotlib.pyplot as plt

images, labels = [], []
for idx, (img, lbl) in enumerate(dataset):
    images.append(img); labels.append(lbl)
    if idx == 15: break

fig, axes = plt.subplots(4, 4, figsize=(8, 8))
for i, ax in enumerate(axes.flat):
    ax.imshow(images[i])
    ax.set_title(f"Class: {labels[i]}")
    ax.axis("off")
plt.show()

Every pixel arrived over HTTP — no local files touched.

Key Takeaways

  1. Seek time dominates when reading millions of small files — GPUs idle at 0%
  2. Contiguous sharding (tar archives) amortizes seek cost across thousands of samples
  3. Object storage (S3/MinIO) is the universal API for remote training data
  4. WebDataset streams tar shards over HTTP directly into your training pipeline
  5. Lazy pipelines avoid materializing data on local disk
